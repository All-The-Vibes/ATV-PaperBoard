02Flow

How a single tool call propagates end-to-end.

5 hops

Same five hops the screenshots above capture. The hook never blocks tool execution — it only injects an <code>additionalContext</code> suggestion that Copilot threads into the agent's next turn.

01
Agent writes a file
Copilot CLI invokes its create tool. The file is on disk before any hook runs.
02
Copilot CLI fires postToolUse
Per hooks.json, our hook command receives the documented JSON payload on stdin: sessionId, toolName, toolArgs.path, toolResult.
03
Hook classifies + suggests
The pure helpers extract the path, the shared SPEC §16 classifier fires Rule 1 (table detection) / Rule 2 (skiplist) / Rule 3 (self-recursion) / Rule 4 (30s cooldown), and the hook prints {"additionalContext": …} to stdout.
04
Copilot threads the suggestion
Copilot CLI injects the additionalContext into the agent's next turn — the analogue of Claude Code's <system-reminder> block. The agent decides whether to run paperboard render.
05
RENDER + COMPOUND pillars run
Render produces the triple in ${COPILOT_HOME}/plugin-data/atv-paperboard/artifacts/ (or the PAPERBOARD_ARTIFACT_DIR override), and the gallery auto-regenerates so cross-harness artifacts land in the same compound index.

04Caveat

One thing Copilot CLI hooks can't do.

by design

Copilot CLI hooks are fail-open.

  • Non-zero exit codes are logged but tool execution continues regardless.
  • So our hook always exits 0 and signals through stdout JSON only.
  • Hard enforcement (blocking on undeclared hex, missing DESIGN.md, etc.) must live in CI — see the .github/workflows/ recipe shipped with the repo.
